Our Commitment to Data Protection

Security at PocketPal

PocketPal is committed to protecting the security and privacy of user data. This document outlines our comprehensive security measures, architecture, and practices.

1. Security Architecture

Our security architecture is designed with multiple layers of protection:

Layer 1: Network Security

  • TLS/SSL encryption for all data in transit
  • Secure network protocols
  • Firewall protection
  • DDoS mitigation

Layer 2: Application Security

  • Server-side validation of all user inputs
  • Role-based access control (RBAC)
  • Authentication and authorization mechanisms
  • Secure session management

Layer 3: Data Security

  • Encryption at rest
  • Database security with Row Level Security (RLS)
  • Secure data storage
  • Data backup and recovery procedures

Layer 4: Infrastructure Security

  • Secure cloud infrastructure (AWS, Supabase)
  • Regular security updates and patches
  • Intrusion detection and monitoring
  • Access controls and logging

2. Authentication & Authorization

2.1 Authentication Methods

Magic Link Authentication
  • Passwordless authentication via secure email links
  • Time-limited, single-use tokens
  • Secure token generation and validation
  • Email verification required for all accounts
JWT Token Verification
  • All authentication tokens verified with Supabase
  • Token expiration and refresh mechanisms
  • Secure token storage
  • Token revocation capabilities

2.2 Authorization Framework

Role-Based Access Control (RBAC)
  • Tech Admin: Full system access, user management
  • School Admin: Student and group management
  • Student: Basic access to student features

2.3 Session Management

  • Secure session handling
  • Session timeout capabilities
  • Automatic logout on security events
  • Session token rotation

2.4 Account Security

  • Email verification required
  • Account lockout after failed attempts
  • Suspicious activity detection
  • Password reset procedures (via magic links)

3. Data Protection

3.1 Encryption

Encryption in Transit
  • TLS 1.2 or higher for all connections
  • HTTPS for all web traffic
  • Encrypted API communications
  • Certificate pinning (where applicable)
Encryption at Rest
  • Database encryption
  • Encrypted backups
  • Encrypted file storage
  • Key management best practices

3.2 Database Security

Row Level Security (RLS)
  • RLS policies on all sensitive tables
  • Users can only access their own data
  • Admin access controlled by policies
  • Policy-based data isolation

4. Security Features

4.1 Role Tampering Prevention

  • Detection of unauthorized role modifications
  • Automatic logout on role mismatch
  • Real-time role verification
  • All role assignments validated server-side

4.2 Security Monitoring

  • Real-time security event logging
  • Intrusion detection systems
  • Anomaly detection
  • Automated alerting

5. Security Best Practices

5.1 Development Security

  • Input validation and sanitization
  • Output encoding
  • Secure error handling
  • Regular code reviews
  • Regular dependency updates
  • Vulnerability scanning

5.2 Operational Security

  • Principle of least privilege
  • Regular access reviews
  • Multi-factor authentication for admins
  • Secure access procedures

6. User Security Responsibilities

Users are responsible for:

  • Keeping email accounts secure (authentication method)
  • Not sharing account credentials
  • Logging out on shared devices
  • Reporting suspicious activity
  • Using strong, unique email passwords
  • Enabling two-factor authentication on email (if available)

  • CCPA/CPRA
  • FERPA (educational context)
  • Industry best practices (OWASP, NIST, ISO 27001 principles)
